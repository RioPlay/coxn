== Laws (do not violate)

From `DESIGN.adoc` and `docs/routing.adoc`:

* *coxn steers and gates; it does not become a provider platform.* Piggyback via
  subprocess + env + home dirs; do not reimplement OAuth, MCP hosts, or thread
  orchestration.
* *Selection is data.* Named `{instance, model}` + `[route]` table; coxn looks up,
  never infers model from prompt.
* *Secrets never in committed config.* Keys in env or `~/.config/coxn/secrets/`
  (mode `0600`), same split as T3's `ServerSecretStore` + redacted settings.
* *Minimal diff.* No drive-by refactors; keep `main.rs` extraction (Phase K) as a
  separate batch unless this work forces a small `app.rs` extract for provider only.
* *Tests hermetic where possible.* Fake subprocess / mock HTTP for probes; live
  network only in manual verification notes.

=== Phase A1 ŌĆö Provider instance types + config load

*New module:* `src/provider.rs` (or `src/provider/mod.rs` if split early).

*Types:*

[source,rust]
----
enum ProviderDriver {
    OpenAiCompat,  // HTTP chat/completions (today's openai.rs)
    Stub,
    // Phase A3: Codex, ClaudeCli ŌĆö behind feature or cfg if desired
}

struct ProviderInstance {
    id: String,           // slug: local, openrouter, codex_work
    driver: ProviderDriver,
    display_name: Option<String>,
    enabled: bool,
    // OpenAiCompat
    base_url: Option<String>,
    // CLI drivers (A3)
    binary: Option<String>,
    home_path: Option<String>,      // CODEX_HOME or Claude HOME
    shadow_home: Option<String>,    // Codex only
    env: Vec<(String, String)>,   // non-secret overrides
    secret_env_keys: Vec<String>,   // names resolved from COXN_KEY_<id> or secrets file
}

struct ModelSelection {
    instance_id: String,
    model: String,
}
----

*Config file* ŌĆö prefer extending `.aden/config.toml` (existing `aden config` seam):

[source,toml]

[provider.openrouter]

[provider.codex_work]

[route]

=== Phase A2 ŌĆö Resolution seam + secrets

*Functions:*

* `resolve_instance(dir, caps, instance_id, model) -> (AnyModel, ModelSel)`
* `resolve_role(dir, caps, role) -> Option<ModelSelection>` ŌĆö parse `instance:model`
* `resolve_active(dir, caps) -> ModelSelection` ŌĆö `route.active` or legacy default
* `secret_for_instance(id) -> Option<String>` ŌĆö order: `COXN_KEY_<ID_UPPER>`, then

*Env merge helper:* `merge_instance_env(instance, base_env) -> HashMap` for future CLI
spawn (A3).

*Wire:* Replace direct `resolve_model` usage in `run_tui`, `drive`, `/execute` with
`resolve_active` / `resolve_role` ŌåÆ `resolve_instance`. Keep `COXN_MODEL_*` as
override: env still wins over config (document in README).

*Cost safety (coxn-specific):*

* `enabled = false` instances are never selected automatically.
* Cloud `openai_compat` instances (`base_url` not localhost/127.0.0.1) require either
  a resolved secret or explicit `COXN_ALLOW_CLOUD=1` ŌĆö otherwise doctor/auth status
  reports blocking, pump stays on stub/local.
* Document: *no background LLM calls in coxn* (unlike T3 text-generation); only
  user-initiated turns and explicit probes in `auth status`.

*Tests:* resolution order, per-instance key, cloud gate, role fallback to active.

*Commit:* `feat(provider): resolve instance:model and cloud opt-in gate`

=== T3-derived gap review after first Track A pass

Source files reviewed in the T3 Code repo:

* `packages/contracts/src/providerInstance.ts`
* `apps/server/src/provider/ProviderInstanceEnvironment.ts`
* `apps/server/src/auth/ServerSecretStore.ts`
* `apps/server/src/provider/Layers/CodexProvider.ts`
* `apps/server/src/provider/Layers/ClaudeProvider.ts`
* `apps/server/src/provider/Drivers/CodexHomeLayout.ts`
* `apps/server/src/provider/Drivers/ClaudeHome.ts`
* `docs/providers/codex.md`, `docs/providers/claude.md`

What this revealed:

* *CLI providers are not just auth rows.* T3 treats Codex/Claude as provider
  drivers with their own runtime adapters and text-generation bridges. coxn's
  first pass can configure `driver = "codex"` / `driver = "claude_cli"`, but it
  still cannot use them for `call_model()` until A6 adds a `CliPiggyback` backend.
* *Codex auth status should use `codex app-server`.* T3 starts `codex
  app-server`, sends `initialize`, `initialized`, then `account/read`, and only
  then reports account type/email/auth state. `codex --version` is an install
  check, not an auth check.
* *Claude/OpenRouter should be modeled as a Claude environment.* T3's
  OpenRouter path is a Claude provider with `ANTHROPIC_BASE_URL`,
  `ANTHROPIC_AUTH_TOKEN`, empty `ANTHROPIC_API_KEY`, and optional
  `ANTHROPIC_DEFAULT_*_MODEL` variables. coxn already parses `env.*`, but cannot
  execute `claude_cli` yet.
* *Provider ids and env names need schema-level validation.* T3 restricts
  provider ids to identifier-like slugs and env vars to real environment
  variable names. coxn now mirrors this in the parser.
* *Unknown drivers must be unavailable, not reinterpreted.* T3 preserves unknown
  driver envelopes and marks them unavailable. coxn now stores
  `ProviderDriver::Unknown` instead of silently falling back to OpenAI-compatible.
* *Secret writes should be atomic and permissioned.* T3 writes secrets through a
  temp file, renames, and applies `0600`; coxn now follows that shape for
  `coxn auth set-key`.

New follow-up phases:

* [x] A7 Codex auth probe: spawn `codex app-server`, JSONL request/response
  `initialize` -> `initialized` -> `account/read`; report account email/type and
  auth status. Use fake binary tests.
* [ ] A8 Claude environment presets: document and example-config an
  OpenRouter-through-Claude profile using `env.ANTHROPIC_*` and a dedicated
  `home_path`; make `auth login` print the exact HOME-scoped commands.
* [x] A9 CLI backend MVP: `AnyModel::CodexPiggyback` for Codex first, text-only
  turns through `codex app-server`, no tool parity. Preserve coxn's approval/gate
  boundary by keeping coxn tools on the coxn side.
* [ ] A10 Claude backend MVP: spawn Claude with per-instance `HOME` and env;
  translate text-only request/reply or use Claude Agent SDK equivalent if a Rust
  seam is available.
* [ ] A11 Status snapshots: cache explicit `auth status` results briefly so
  `/model` and status lines can show instance readiness without background cloud
  probes.
