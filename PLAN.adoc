prerequisites are out of scope here and tracked under Phase 2 blockers.

== Phase 2 (coxn-side): aden wiring

coxn drives aden as a subprocess via `std::process` (no new deps; DESIGN
allowed linking aden-core OR shelling out, and the 3-dep rule forces the
latter). Tested against the offline aden branch via `COXN_ADEN_BIN`.

* [x] P2.1 `aden` seam: `std::process` shell-out -- `gate` (exit-code -> `GateVerdict` + message), `scope` (manifest JSON), `pull` (asm/understand). `COXN_ADEN_BIN` override. 3 fake-aden tests (verdict decode, closed-gate-on-spawn-fail, text/error).
* [x] P2.2 Pull-context tools: `AsmTool`/`UnderstandTool` (aden_asm/aden_understand) registered in main's registry, rooted at the working dir; model pulls on demand. Tests for names + empty-arg rejection.
* [x] P2.3 Gate enforcement: `Gate` trait + `AdenGate` (wraps the seam); pump gates mutating tools (`Tool::mutates`), blocks non-in-scope edits + feeds the verdict back to the model + records the block; main raises the confirm modal. 3 pump tests (block/allow/non-mutating). Live LLM-driven edit awaits Phase 3.
* [x] P2.4 Scope->context loader: `COXN_TASK_NAME`/`SEEDS`/`BUDGET` -> `aden scope` defines the mandate (manifest persisted -> gate) and `Pump::set_context` loads exactly the seeds' `aden asm` context. NO serde -- coxn loads from its own seed inputs + the manifest file, parses nothing.
* [x] P2.5 Savings: `aden::savings` lifts the all-time figure from `aden status` (pure `extract_savings`, tested); status line shows it, refreshed after each turn. Falls back to base text when aden is absent.
* [x] P2.6 Deferred tool discovery: `aden_tools` seam searches a latent catalog by intent; `names()` advertises only the seam + active tools (no bloat), latent tools dispatchable once known. main registers the aden tools latent. 3 tests. (Full schema-into-next-turn registration is a Phase 3 refinement with a real provider's tool list.)

== Phase 4 (coxn-side): the harness -- gated editing  -- DONE

The action surface that makes coxn drive (not suggest). All verified live end
to end against a throwaway git repo + LM Studio gpt-oss-20b, and the aden seam
verified against a real third-party codebase.

* [x] P4.1 Action tools: `edit` (replace an exact unique string) and `write_file` (create/overwrite, makes parent dirs). Both `mutates() == true` and declare `target_path` so the pump can snapshot.
* [x] P4.2 Apply -> gate -> revert: per DESIGN, aden reads the working-tree diff, so the pump applies the edit, runs `impact-diff --scope`, and on scope-escape/blast-leak reverts only that file (restore bytes / remove a new file) and feeds the verdict back. In-scope edits persist. No git, no new deps. Unit-tested (revert/persist/no-gate) + live.
* [x] P4.3 Editing requires a scope: no task -> no gate -> mutating tools refused (never ungated), and the action tools are registered only when the scope actually gated.
* [x] P4.4 Task-mode operating preamble (`AGENT_PREAMBLE`): one sanctioned nudge so a weak model with the empty default prompt actually calls the action tools (gpt-oss-20b otherwise answers "I can't edit files"). Added only when gated.

* [x] P4.6 README.md: quickstart, model selection, task mode, slash commands, the aden seam.
